# Assistant Slice 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `subagent-driven-development` (recommended) or
> `executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for
> tracking.

**Goal:** Build the first authenticated assistant slice: authenticated root redirect, assistant
routes, persisted sessions/messages/runs/events, per-session model selection, session management UI,
async turn execution, inline debug detail, and a real `auggie` backend behind a shared abstraction.

**Architecture:** ExClaw owns the canonical transcript and operational records. A shared backend
contract plus a model catalog isolate backend-specific logic, with `auggie` as the first real
adapter and a fake backend used in tests. LiveView renders session and transcript state from the
Assistant context, starts runs through an application-level runner, and refreshes from PubSub
notifications so in-flight runs survive UI timing differences.

**Tech Stack:** Phoenix 1.8 LiveView, Ecto + SQLite, Task.Supervisor + Phoenix.PubSub,
`System.cmd/3` for `auggie`, ExUnit + Phoenix.LiveViewTest.

## Execution Strategy

- This plan is dependency-ordered and intentionally **not** a 1:1 mapping from PRD user stories.
- Use TDD for each task: write/adjust focused tests first, run them red, implement minimally, rerun green.
- Use a fake backend in tests from the start so UI/context tests never depend on real `auggie` availability.
- Keep the documented contracts stable even when scaffolding temporarily precedes the final real
  backend wiring.
- After each task-sized checkpoint, the agent should present the changes and verification results,
  ask the user to review/commit, and wait for explicit approval before proceeding.

## Documentation Update Policy During Execution

- The PRD should be treated as mostly stable during implementation.
- Update the design doc or implementation plan during execution only when:
  - implementation reveals a previously wrong assumption,
  - a contract or schema detail needs clarification,
  - backend-specific technical reality needs to be recorded,
  - or a handoff/safety note is needed for later tasks.
- Do not silently change approved product behavior, scope, or acceptance criteria during implementation.
- If implementation suggests a meaningful product-scope or UX change, stop and ask the user before
  changing the PRD.
- When doc updates are made during implementation, mention them explicitly in the next review checkpoint.

    - `{:error, reason}` on failure
  - Define normalized `model_info` entries with at least:
    - `id`
    - `display_name`
    - optional extra fields such as `description`, `context_window`, capability flags, and `metadata`
  - Keep app-level default-model selection outside the backend-returned model list.
  - Define a normalized `run_turn/1` request shape that includes:
    - `run_id`
    - `session_id`
    - `model`
    - normalized `messages`
    - optional `backend_session_id`
    - `workspace_root`
    - optional future-facing `stream?` and `metadata`
  - Do not include `backend` in the `run_turn/1` request shape; the backend module has already been
    selected before the callback is invoked.
  - Define normalized `messages` as ordered oldest-to-newest entries with the shape:
    - `%{role: :user | :assistant, content: String.t()}`
  - In slice 1, keep `messages` limited to `:user` and `:assistant` roles and exclude DB/UI
    metadata from the payload.
  - Treat any conversion from this normalized payload into backend-specific JSON or CLI input as the
    adapter's responsibility.
  - When building `run_turn/1` requests, derive `backend_session_id` from the most recent prior run
    in the same session that has a non-null backend session ID, even if that prior run failed.
  - Do not add a cached backend-session-id field to `assistant_sessions` in slice 1.
  - Define a normalized backend result shape that can carry:
    - `{:ok, success_result}` on success
    - `{:error, failure_result}` on failure
  - Make `success_result.reply_text` required and treat it as the value used to persist the
    assistant transcript message.
  - Allow both success and failure results to carry:
    - backend session/run IDs
    - request/response snapshots
    - events
  - Require `failure_result` to carry at least:
    - `error_type`
    - `error_message`
